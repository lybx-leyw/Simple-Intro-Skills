# Intro — Self-Assessment Framework

你是同一个模型，在评估**你自己刚刚在本对话中产生的输出**。追加一次诚实的、基于证据的质量自评。**不自动改写**——只评估。

## 触发

`/Intro` · `<query> /Intro` · `<query> 自评/自省/自我评估` · `自评你刚才的<type>`

## 原则

- **诚实 > 讨好**。60 分真实 > 90 分虚假
- **证据必须具体**。引用输出的原文/原代码，禁止 "overall feels like 7"
- **权重不可调**。按类型使用固定权重
- **百分制**。维度 0-10 分 × 权重 → 加权和 → /100

## 第一步：判定类型（按优先级）

1. 含可执行代码 → **代码**
2. 含推理/论证/比较 → **分析推理**
3. 翻译 → **翻译**
4. 故事/文章/诗歌 → **创意写作**
5. 事实性问答 → **事实问答**
6. 不确定 → **通用/混合**，注明原因

## 第二步：评分

**通用锚定**：9-10=专业级 · 7-8=良好有小疵 · 5-6=及格有不足 · 3-4=明显缺陷 · 1-2=严重错误 · 0=全错/缺失

### 维度速查（按类型）

| 类型 | D1 (权重) | D2 (权重) | D3 (权重) | D4 (权重) | D5 (权重) |
|------|----------|----------|----------|----------|----------|
| **事实问答** | 准确性 30% | 完整性 25% | 清晰度 20% | 简洁性 15% | 来源意识 10% |
| **创意写作** | 连贯性 25% | 语言 25% | 创意 20% | 提示遵循 20% | 情感冲击 10% |
| **代码** | 正确性 35% | 效率 20% | 可读性 20% | 健壮性 15% | 文档 10% |
| **翻译** | 语义准确 40% | 自然度 30% | 术语 15% | 文化适配 15% | — |
| **分析推理** | 逻辑 30% | 深度 25% | 证据 20% | 公正性 15% | 组织 10% |
| **通用/混合** | 任务完成 35% | 质量 30% | 清晰度 20% | 局限意识 15% | — |

### 各维度简要标准

**准确性**：事实对否？有无错误？高分=经得起验证。**完整性**：覆盖全面否？有无遗漏？**清晰度**：结构分明、易懂？**简洁性**：无冗余重复？**来源意识**：区分了事实和推测？

**连贯性**：叙事流畅、结构合理？**语言**：用词句式修辞出色？**创意**：新颖还是模板化？**提示遵循**：完全遵循用户要求？**情感冲击**：打动读者？

**正确性**：逻辑对、边界处理？**效率**：复杂度合理、无冗余计算？**可读性**：命名注释结构清晰？**健壮性**：错误处理、输入校验？**文档**：必要的注释/README？

**语义准确**：忠实原文、无漏增错译？**自然度**：像目标语言原生表达？**术语**：准确且一致？**文化适配**：习语典故适当转换？

**逻辑**：推理链完整无漏洞？**深度**：触及核心还是表面？**证据**：主张有支撑？**公正性**：考虑反面、无偏见？**组织**：层次结构清晰？

**任务完成**：完成了所有要求？**质量**：整体综合水平？**局限意识**：指出了不确定性？

## 第三步：输出自评

```
### Intro Self-Assessment
**Output Type:** <类型>  **Overall:** XX/100 — <一句话定性>

| Dimension | Wt | Score | Evidence |
|-----------|---|-------|----------|
| <名称> | <N>% | X/10 | "<输出中一句原文引用>" |
| ... | ... | ... | ... |
**Aggregated: XX/100**

**Critical Issues:**
- **[Critical]** <会导致误导或不可用的问题，引用原文>  *(无则写 None)*
- **[Minor]** <影响质量但不致命>  *(最多 3 条，无则写 None)*

**Strengths:**
- <引用原文的优点，1-3 条>
```

### 格式规则

- **Evidence 列**：引用输出原文，≥5 词。禁止 "see output" / "overall..."
- **Critical** 定义：事实错误 · 安全漏洞 · 逻辑崩塌 · 完全不可用。一般性不足一律归 Minor。
- 无 Critical ≠ 高分。可 "无致命错误但整体平庸"（60-65）。
- Strengths 至少 1 条，最多 3 条。

## 分数段

| 分数 | 含义 |
|------|------|
| 90-100 | 卓越，几乎无需修改 |
| 75-89 | 优秀，小幅优化 |
| 60-74 | 良好，可用但有改进方向 |
| 40-59 | 需改进，明显不足 |
| 20-39 | 差，多处关键问题 |
| 0-19 | 极差，基本不可用 |

## 边缘情况

| 场景 | 处理 |
|------|------|
| 输出太短 (<3 句 / <10 行代码) | 仍评估，注明 "输出过短"，相关维度标 N/A |
| 敏感内容 | 跳过，注明原因 |
| 无前置输出 | 提示用户先提供内容 |
| 类型模糊 | 通用/混合，注明 ambiguous |
| fix it | 参考 Critical Issues 产出修订版 |
| 纯代码无文字 | 文档维度=0，注明原因 |
| 用户指定某维度严查 | 该维度权重 ×2，注明 per user |
| 评估与用户直觉相反 | 不辩解，注明 "用户观感可能不同，评分标准下..." |
